# References — Checklist (как разбирать внешний проект)

Цель: быстро понять, что можно взять **сейчас** (MVP), что — **потом**, и не втащить лицензионные/архитектурные проблемы.

## 1) Лицензия (обязательно)
- [ ] Лицензия найдена (LICENSE / header).
- [ ] Понятно, можно ли копировать код в MIT‑проект:
  - MIT/Apache/BSD → обычно ок (с атрибуцией).
  - GPL/AGPL → **код не копировать**, только идеи/паттерны.
- [ ] Зафиксировать вывод в заметке по проекту (`License / usage notes`).

## 2) Релевантность (что мы ищем)
- [ ] Есть функции, похожие на наши (editor/selection/undo/render/paste/plain text/backup/search).
- [ ] Понятно, какие 2–3 “узких места” проекта нам важны.

## 3) Где смотреть (точечно)
- [ ] Найдены ключевые файлы/модули (пути/названия).
- [ ] Найдены точки входа (init/editor loop/handlers).

## 4) Что извлекаем (в виде паттернов, не копипасты)
- [ ] Алгоритм/подход (в 3–10 буллетах).
- [ ] UX‑решение (ожидаемое поведение, крайние случаи).
- [ ] Риски/ограничения (что может сломаться у нас).

## 5) Два слоя: MVP Now / Later
- [ ] Выписать “MVP Now”: что применимо прямо сейчас, без перестройки архитектуры.
- [ ] Выписать “Later”: идеи/фичи на будущее (без кода, без подготовки в текущем MVP).

## 6) Итог
- [ ] Создан файл заметок в `docs/references/projects/…`
- [ ] Добавлена строка в `docs/references/INDEX.md`
- [ ] Если вывод влияет на “правила проекта” — обновить `docs/CONTEXT.md`
